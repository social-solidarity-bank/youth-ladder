# 청년 희망사다리 심사 화면 설계 v0.2

## 목표

관리자 페이지를 단순 신청서 뷰어에서 심사 워크벤치로 확장한다. 사업담당자는 1차 서류심사와 보완요청을 처리하고, 최종심사는 사업담당자와 팀장이 모든 케이스를 함께 검토해 합의 판정을 남긴다. 최종 선정 결과는 품의에 바로 활용할 수 있도록 점수, 판정, 감액 조정, 요약문 형태로 기록되어야 한다.

## 심사 운영 흐름

### 1. 접수 및 신규 분류

- 신청서가 접수되면 기본 상태는 `신규접수`로 둔다.
- 담당자는 목록에서 신규 신청자를 확인하고 1차 서류심사를 시작한다.
- 접수 목록은 상태, 신청금액, 제출일, 보완요청 여부, 최종심사 대상 여부로 빠르게 필터링할 수 있어야 한다.

### 2. 1차 서류심사

- 1차 서류심사는 사업담당자가 수행한다.
- 담당자는 자격요건, 신청서 완성도, 기본 부채·소득 정보, 대환 필요성, 증빙 필요 항목을 확인한다.
- 1차 판정은 `1차검토중`, `보완요청`, `1차적격`, `1차부적격`으로 둔다.
- 1차 서류심사 결과 1차적격으로 분류된 인원을 최종심사 대상으로 넘긴다.

### 3. 보완요청 및 연락

- 보완요청은 1차 서류심사 중 담당자가 생성한다.
- 보완요청에는 요청 항목, 요청 사유, 요청일, 회신기한, 연락 방식, 처리 상태를 남긴다.
- 실제 문자 발송 기능은 넣을 수 있으나 유료 외부 서비스 연동이 필요하므로 1단계에서는 연락 템플릿 생성과 연락 기록 저장을 우선한다.
- 2단계에서 문자 API 또는 알림톡 API를 연결해 실제 발송까지 자동화한다.
- 모든 연락은 품의와 사후 확인을 위해 기록으로 남긴다.

### 4. 최종심사

- 최종심사는 사업담당자와 팀장 2인이 함께 모든 케이스를 검토한다.
- 개별 심사자가 따로 점수를 입력해 평균을 내는 구조가 아니라, 함께 보고 합의한 최종 점수와 판정을 남기는 구조다.
- 최종심사 화면은 신청자별 핵심정보, 1차 검토 메모, 보완요청 이력, 점수표, 합의 판정, 감액 조정 입력을 한 화면에서 볼 수 있어야 한다.

### 5. 최종 선정 및 품의

- 최종 선정은 점수 기반 판정으로 정리한다.
- 80점 이상은 승인, 70점대는 조건부 승인, 70점 미만은 탈락을 기본 기준으로 둔다.
- 조건부 승인일 때는 신청금액 대비 승인금액을 감액 조정할 수 있어야 한다.
- 품의용 요약문은 기존 양식에 맞춰 자동 생성한다.
- 미선정 사유는 별도 분리 화면으로 만들지 않고, 판정 사유와 심사 메모 안에 기록한다.

## 화면 구조

### 1. 좌측 접수 목록

- 기존 신청자 목록을 유지한다.
- 목록 카드에 접수번호, 이름, 신청금액, 접수일, 현재 심사상태, 보완요청 여부를 표시한다.
- 검색은 이름, 접수번호, 휴대전화 뒤 4자리 기준으로 동작한다.
- 필터는 전체, 신규접수, 1차검토중, 보완요청, 1차적격, 1차부적격, 최종심사대상, 승인, 조건부승인, 탈락으로 둔다.

### 2. 상단 요약 바

- 선택된 신청자의 접수번호, 성명, 생년월일, 신청금액, 접수일, 현재 상태를 표시한다.
- 우측에는 주요 상태 변경 버튼을 둔다.
- 상태 변경 시 담당자 메모 또는 변경 사유를 입력하게 한다.
- 상태 변경 기록은 심사 이력에 자동으로 남긴다.

### 3. 본문 탭

- 신청서 요약: 기본정보, 자격요건, 신청금액, Q1~Q3 핵심 서술
- 재무검토: 부채 목록, 대환 대상, 월소득, 월지출, 순가처분소득, DSR
- 1차 서류심사: 서류검토 체크리스트, 보완요청, 담당자 메모
- 최종심사: 점수표, 합의 판정, 승인금액, 품의용 요약
- 심사이력: 상태 변경, 보완요청, 연락 기록, 메모 저장 이력

### 4. 우측 심사 패널

- 현재 상태와 다음 조치를 항상 표시한다.
- 1차 서류심사 단계에서는 보완요청 생성, 1차적격 처리, 1차부적격 처리를 지원한다.
- 최종심사 단계에서는 점수 입력, 판정 선택, 승인금액 입력, 품의용 요약 생성 버튼을 지원한다.
- 저장 버튼은 상태, 점수, 판정, 승인금액, 메모 중 변경된 항목을 저장한다.

## 1차 서류심사 기능

### 체크 항목

- 기본 자격요건 충족 여부
- 신청서 필수 항목 완성도
- 부채 내역 입력 충실도
- 소득·지출 정보 입력 충실도
- 대환 필요 채무 확인 가능 여부
- 서술 답변의 구체성
- 추가 증빙 필요 여부

### 1차 판정

- `1차검토중`: 담당자가 검토 중
- `보완요청`: 신청자에게 추가 자료 또는 내용 보완 요청
- `1차적격`: 최종심사 대상으로 이동
- `1차부적격`: 최종심사 대상에서 제외

### 보완요청 관리

- 요청 항목은 복수 선택 가능해야 한다.
- 요청 사유와 안내 문구를 저장한다.
- 회신기한을 지정할 수 있어야 한다.
- 연락 기록에는 전화, 문자, 이메일, 기타를 남긴다.
- 실제 발송 전 단계에서는 복사 가능한 안내문을 제공한다.

## 최종심사 기능

### 점수표

- 최종심사는 판정 중심이지만 품의 기록을 위해 점수표를 둔다.
- 점수는 합의 점수 1개를 저장한다.
- 세부 항목별 점수와 총점을 모두 남긴다.
- 총점 기준은 승인 80점 이상, 조건부 승인 70점대, 탈락 70점 미만으로 둔다.

### 점수 항목 초안

- 지원 필요성
- 상환 가능성
- 대환 효과
- 신청서 진술의 구체성
- 보완자료 신뢰도
- 사업 취지 적합성

### 판정

- `승인`: 신청금액 또는 조정금액 승인
- `조건부승인`: 승인하되 감액 또는 추가 조건을 붙임
- `탈락`: 최종 미선정

### 조건부 승인

- 조건부 승인 시 승인금액을 신청금액보다 낮게 입력할 수 있어야 한다.
- 감액 사유는 심사 메모와 품의용 요약문에 반영한다.
- 조건부 승인 상태에서는 승인금액 입력을 필수로 한다.

## 품의용 요약 자동 생성

### 목적

- 기존 품의 양식에 붙여넣을 수 있는 요약문을 자동 생성한다.
- 담당자가 매번 신청서 내용을 다시 요약하는 시간을 줄인다.

### 포함 항목

- 신청자 기본 요약
- 신청금액과 최종 승인금액
- 주요 부채 상황
- 월소득·월지출·상환 가능성 요약
- 대환 필요성과 기대 효과
- 최종 점수와 판정
- 조건부 승인 또는 감액 조정 사유
- 담당자·팀장 합의 기록

### 생성 방식

- 1단계는 템플릿 기반 자동 문장 조합으로 구현한다.
- 2단계에서 필요하면 AI 요약을 검토한다.
- 생성된 요약문은 담당자가 수정 후 저장할 수 있어야 한다.

## 문자·알림 발송

### 가능 여부

- 실제 문자 발송은 가능하다.
- 다만 외부 문자 발송 서비스 가입, 발신번호 등록, API 키 관리, 발송비 충전 또는 후불 정산이 필요하다.
- 운영 초기에는 실제 발송보다 연락 템플릿 생성과 연락 기록 저장을 먼저 구현한다.

### 비용 기준

- 문자 발송 서비스는 보통 월 기본료 없이 건당 과금 구조를 제공한다.
- SMS, LMS, 알림톡은 메시지 길이와 채널에 따라 단가가 다르다.
- 보완요청은 내용이 길어질 가능성이 높으므로 SMS보다 LMS 또는 알림톡 템플릿 방식이 현실적이다.

### 운영 단계

- 1단계: 안내문 생성, 복사 버튼, 연락 기록 저장
- 2단계: 담당자가 화면에서 발송 버튼을 누르면 실제 문자 발송
- 3단계: 보완요청 생성 시 자동 발송까지 확장

## 데이터 모델 제안

기존 `applications` 테이블은 접수 원문과 최소 상태만 유지한다.

- 현재 심사상태
- 1차 판정
- 최종 판정
- 신청금액
- 최종 승인금액
- 마지막 수정 시각

접수 원문 `form_data`에는 신청자가 입력한 전체 신청서 값을 누락 없이 보관한다.

- KCB 조회와 본인확인을 위해 주민등록번호 앞자리, 뒷자리, 전체번호를 함께 저장한다.
- 단기 운영 단계에서는 기존 테이블 구조 변경 없이 `form_data` 안에 저장한다.
- 장기적으로는 주민등록번호를 별도 보호 컬럼 또는 암호화 저장 구조로 분리한다.

심사 기록은 별도 테이블로 분리한다.

- 심사 단계
- 판정
- 점수
- 사유 코드
- 메모
- 저장한 사용자
- 저장 시각

보완요청 기록도 별도 테이블로 분리한다.

- 요청 항목
- 요청 사유
- 안내문
- 회신기한
- 연락 방식
- 처리 상태
- 처리 시각

품의용 요약문도 별도 기록으로 남긴다.

- 자동 생성 원문
- 담당자 수정본
- 최종 품의 반영본
- 생성 시각
- 수정자

## 저장 규칙

- 상태 변경과 메모 저장은 모두 Supabase Auth 세션이 있을 때만 가능하다.
- 관리자 테이블에 등록된 사용자만 심사 정보를 조회하고 저장할 수 있다.
- 1차부적격, 보완요청, 조건부승인, 탈락은 메모 입력을 필수로 한다.
- 조건부승인은 승인금액 입력을 필수로 한다.
- 최종심사 저장 시 점수, 판정, 승인금액, 품의용 요약문을 함께 저장한다.
- 신청자 원문 데이터는 수정하지 않고, 심사 데이터는 별도 기록으로 남긴다.

## 접수 데이터 정합성 원칙

- 접수 화면에 실제로 존재하는 입력값만 `form_data`의 현재 저장 기준으로 삼는다.
- 화면에서 삭제된 질문은 관리자 상세 화면의 고정 항목과 CSV 기본 헤더에서도 제거한다.
- 과거 접수건에 실제 값이 남아 있는 삭제 항목은 원문 데이터 또는 CSV 추가 컬럼으로만 확인한다.
- 사용자가 입력할 수 있는 동적 항목도 누락 없이 저장한다. 현재 기준에는 거주비용 상세, 정부지원 월금액 상세, 채무조정 회차, 위기 상황 대처 계획이 포함된다.
- 관리자 리포트에는 값이 없는 선택·구버전 항목을 빈 행으로 표시하지 않는다.
- 새 질문을 추가하거나 기존 질문을 삭제할 때는 `index.html` 입력 항목, `collectData()` 저장 키, Supabase 저장 컬럼, `admin.html` 표시 항목, CSV 헤더를 함께 점검한다.

## 보안 및 개인정보

- 브라우저 저장소에는 신청자 개인정보와 심사 메모를 저장하지 않는다.
- `applications`는 RLS 정책으로 보호한다.
- 익명 사용자는 신청서 제출만 가능하고 조회, 수정, 삭제는 할 수 없어야 한다.
- 관리자 조회는 관리자 테이블에 등록된 인증 사용자만 허용한다.
- 관리자 화면은 필요한 컬럼만 조회하되, 심사 담당자가 접수값 누락 여부를 확인할 수 있도록 접수 원문 확인 영역과 CSV 내보내기를 제공한다.
- KCB 조회용 주민등록번호는 관리자 화면의 별도 영역과 CSV에만 표시한다.
- `form_data` 원문은 접근 권한을 좁히고, 주민등록번호는 장기적으로 최소 저장 또는 별도 보호 구조를 검토한다.
- 상태 변경, 메모 저장, 보완요청, 연락 기록, 품의용 요약 수정은 모두 감사 로그를 남긴다.
- 출력 또는 PDF 저장 화면에는 주민등록번호 원문을 노출하지 않는다.

## CSV 내보내기

- 담당자가 현재 접수 전체를 대조할 수 있도록 전체 접수 원문 CSV를 제공한다.
- KCB 조회 준비를 위해 접수번호, 성명, 휴대전화, 생년월일, 주민등록번호 앞자리, 주민등록번호 뒷자리, 주민등록번호 전체, 저장 상태를 포함한 CSV를 제공한다.
- 기존 접수건 중 주민등록번호 뒷자리가 저장되지 않은 건은 `뒷자리 미저장`으로 표시해 보완 요청 대상을 선별할 수 있게 한다.
- CSV에는 개인정보와 고유식별정보가 포함되므로 관리자 확인 경고를 거친 뒤 내려받게 한다.

## 구현 우선순위

1. 1차 서류심사 상태값과 보완요청 기록 구조 추가
2. 관리자 화면에 1차 서류심사 패널 추가
3. 보완요청 안내문 생성과 연락 기록 저장 기능 추가
4. 최종심사 점수표와 합의 판정 저장 기능 추가
5. 조건부 승인 시 승인금액 감액 조정 기능 추가
6. 품의용 요약문 자동 생성 기능 추가
7. KCB 조회용 주민등록번호 저장과 전체 접수 원문 CSV 내보내기
8. 문자 또는 알림톡 실제 발송 연동 검토
9. 관리자 조회 최소화와 심사 기록 분리

## 미결 쟁점

- 1차부적격을 시스템상 최종 탈락과 분리할지 여부
- 최종심사 점수 항목별 배점
- 조건부 승인 감액 기준
- 보완요청 회신기한 기본값
- 문자 발송 서비스 선택과 비용 부담 방식
- 품의용 요약문의 실제 기존 양식 반영 방식
- 주민등록번호의 장기 저장 방식과 암호화 적용 범위
