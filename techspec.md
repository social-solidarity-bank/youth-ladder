# 청년 희망사다리 심사 화면 설계 v0.1

## 목표

관리자 페이지를 단순 신청서 뷰어에서 심사 워크벤치로 확장한다. 심사자는 신청자별 핵심 위험과 보완 필요 항목을 한 화면에서 확인하고, 판정, 사유, 메모, 다음 조치를 저장할 수 있어야 한다.

## 화면 구조

### 1. 좌측 접수 목록

- 기존 신청자 목록을 유지한다.
- 목록 카드에 접수번호, 이름, 신청금액, 접수일, 현재 심사상태를 표시한다.
- 검색은 이름, 접수번호, 휴대전화 뒤 4자리 기준으로 동작한다.
- 필터를 추가한다: 전체, 신규접수, 보완요청, 1차적격, 부적격, 최종심사대상.

### 2. 상단 요약 바

- 선택된 신청자의 접수번호, 성명, 생년월일, 신청금액, 접수일, 현재 상태를 표시한다.
- 우측에는 상태 변경 버튼을 둔다: 보완요청, 1차적격, 부적격, 최종심사대상.
- 상태 변경 시 사유 입력 모달을 먼저 띄우고 저장한다.

### 3. 본문 탭

- 신청서 요약: 기본정보, 자격요건, 신청금액, Q1~Q3 핵심 서술
- 재무검토: 부채 목록, 대환 대상, 월소득, 월지출, 순가처분소득, DSR
- 심사기록: 판정, 사유 코드, 심사 메모, 보완 요청 내역, 통화 기록
- 제출서류: 요청 서류, 수신 여부, 검토 결과, 미비 사항

### 4. 우측 심사 패널

- 판정 선택: 검토중, 보완요청, 적격, 부적격, 최종심사대상
- 사유 코드: 자격불충족, 소득확인필요, 부채증빙필요, 상환능력우려, 서술보완필요, 기타
- 심사 메모: 내부용 장문 메모
- 다음 조치일: 날짜 선택
- 저장 버튼: 심사 기록 저장

## 데이터 모델 제안

기존 `applications` 테이블에 최소 필드를 추가한다.

- `review_status`: 현재 심사 상태
- `review1_judgment`: 1차 심사 판정
- `review1_reason_codes`: 판정 사유 코드 배열
- `review1_memo`: 내부 심사 메모
- `requested_docs`: 보완 요청 서류 목록
- `next_action_date`: 다음 확인 예정일
- `reviewer_id`: 저장한 관리자 사용자 ID
- `reviewed_at`: 마지막 심사 저장 시각
- `updated_at`: 마지막 수정 시각

심사 이력 추적을 위해 별도 테이블을 권장한다.

- `application_review_logs`
- 주요 필드: `id`, `application_id`, `actor_id`, `from_status`, `to_status`, `reason_codes`, `memo`, `created_at`

## 저장 규칙

- 상태 변경과 메모 저장은 모두 Supabase Auth 세션이 있을 때만 가능하다.
- 저장 전 `review_status`와 변경 사유 중 하나 이상은 필수로 받는다.
- 부적격, 보완요청 판정은 사유 코드 1개 이상을 필수로 받는다.
- 신청자 원문 데이터는 수정하지 않고, 심사 데이터는 별도 필드와 로그에만 남긴다.

## 보안 및 개인정보

- 브라우저 저장소에는 신청자 개인정보와 심사 메모를 저장하지 않는다.
- 관리자 조회는 RLS 정책으로 인증된 관리자만 허용한다.
- 상태 변경, 메모 저장, 서류 검토는 모두 감사 로그를 남긴다.
- 출력 또는 PDF 저장 화면에는 주민등록번호 원문을 노출하지 않는다.

## 구현 우선순위

1. 관리자 상세 화면에 우측 심사 패널 추가
2. `review_status`, `review1_judgment`, `review1_memo` 저장 기능 추가
3. 목록 필터와 상태 배지 추가
4. 보완 요청 서류 체크리스트 추가
5. 심사 변경 로그 테이블 연동

## 미결 쟁점

- 실제 Supabase 테이블 스키마와 RLS 정책 확인 필요
- 심사 상태 코드의 최종 명칭 합의 필요
- 보완 요청 알림을 화면 기록만 할지, 문자·이메일 발송까지 연결할지 결정 필요
- 최종심사와 약정 실행 단계까지 같은 화면에서 다룰지 별도 화면으로 분리할지 결정 필요
